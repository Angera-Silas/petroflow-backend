====================================================
5. DATABASE GENERATION RULE: HIBERNATE VS SQL SCRIPTS
====================================================

This project uses Spring Boot + Hibernate/JPA, and Hibernate is responsible for generating the baseline tables, columns, and standard entity relationships from the Java entities.

Therefore, do not treat SQL migrations as the primary tool for creating all base application tables.

Assume:
- entities define the core tables
- JPA/Hibernate creates the baseline structure
- SQL migrations are mainly for enhancement, optimization, enforcement, reporting support, and production hardening

So do NOT generate SQL scripts that unnecessarily duplicate Hibernate-managed table creation unless there is a strong technical reason.

In this project, SQL scripts and migrations should focus primarily on:
- indexes
- composite indexes
- partial indexes
- unique indexes
- advanced constraints
- triggers
- SQL functions
- stored procedures only where truly justified
- views
- materialized views
- partitioning
- audit helpers
- reporting optimization
- schema enhancement after Hibernate table creation
- DB-side recalculation and consistency helpers

# 1. The big communication flow

For a normal request such as:

**вЂњCreate a sale for tenant ABC station branch 1вЂќ**

the communication path will be:

`HTTP Request`
в†’ `security.jwt.JwtAuthenticationFilter`
в†’ `tenancy.web.TenantContextFilter`
в†’ `controller`
в†’ `dto.request`
в†’ `validation`
в†’ `service`
в†’ `repository`
в†’ `entity`
в†’ `database`
в†’ `mapper`
в†’ `dto.response`
в†’ `common.api.ApiResponse`
в†’ `HTTP Response`

That is the core communication model of the whole backend.

For a scheduled task such as:

**вЂњGenerate yesterdayвЂ™s branch sales summariesвЂќ**

the path becomes:

`jobs.reporting.DailySalesAggregationJob`
в†’ `modules.reporting.service.ReportingService`
в†’ `modules.sales.repository`
в†’ `modules.reporting.query`
в†’ `database`
в†’ `stored summaries / report output`

For onboarding a new tenant:

`platform.organization.controller.OrganizationController`
в†’ `platform.organization.service.OrganizationService`
в†’ `tenancy.lifecycle.TenantOnboardingService`
в†’ `tenancy.schema.TenantSchemaProvisioningService`
в†’ `tenancy.migration.TenantMigrationService`
в†’ `tenancy.bootstrap.TenantAdminBootstrapper`
в†’ tenant becomes usable

That is how the system talks to itself.

17. MULTI-CLIENT DELIVERY MODEL
====================================================

This backend is the central brain for all PetroFlow clients:
- mobile apps built with Flutter
- web apps built with React
- desktop apps built with Flutter, Tauri, Electron, or other clients
- third-party integrations and future partner systems

That means the backend must expose contracts that are:
- stable
- well-versioned
- tenant-aware
- bandwidth-conscious
- offline-friendly where needed
- secure across browser and non-browser clients

API delivery model:

1. REST remains mandatory for:
- authentication and refresh token flows
- file upload and download
- webhooks
- health, readiness, and operational endpoints
- bulk import/export
- offline sync ingestion endpoints
- admin actions with explicit command semantics

2. GraphQL is the preferred contract for:
- dashboards
- mobile app aggregated screens
- branch overview screens
- reporting filters and nested reads
- profile and settings pages that need composed data from multiple modules
- low-chattiness data fetching for mobile and desktop clients

3. WebSocket or SSE is optional and should be introduced only for:
- live dashboards
- notification streams
- job progress events
- sync progress visibility

4. Every client-facing contract must carry:
- trace ID / request correlation
- tenant identity
- authenticated actor identity
- consistent error envelope
- pagination and filtering rules
- idempotency support for critical mutations

GraphQL and REST are not competing styles here.
They are complementary:
- REST for operational commands and infrastructure-style endpoints
- GraphQL for client-driven read models and selected composed mutations

18. FILE AUTHORING STANDARD FOR EVERY FILE CREATED
====================================================

Every new file must be created with an explicit purpose and a predictable shape.
We are not creating random files or dumping logic into generic helpers.

Allowed module subpackages by convention:
- `controller`
- `graphql`
- `dto.request`
- `dto.response`
- `dto.summary`
- `entity`
- `repository`
- `service`
- `service.impl`
- `mapper`
- `validation`
- `query`
- `event`
- `policy`
- `specification`

Each file type must follow these rules:

A. Controller files
- expose REST endpoints only
- accept request DTOs and return response DTOs or `ApiResponse`
- never return JPA entities directly
- never contain business rules beyond request orchestration
- never call repositories directly
- handle idempotency headers where required
- delegate authorization to Spring Security / service policies

B. GraphQL files
- live under module-local `graphql` packages and `src/main/resources/graphql/`
- schema files should be split by bounded context, for example:
  - `graphql/platform/user.graphqls`
  - `graphql/modules/sales.graphqls`
  - `graphql/modules/reporting.graphqls`
- resolver/controller classes only map GraphQL input to service calls
- field resolvers are allowed for composed fields only
- use `DataLoader` for child collections and cross-module lookups to prevent N+1 queries
- authorization remains enforced at service/policy layer, not only in GraphQL annotations

C. Request DTO files
- represent external input contracts only
- include bean validation annotations
- use explicit field names, never overloaded generic maps
- support client timestamps only where business rules require them
- must be serializable and stable for Flutter/React/desktop clients

D. Response DTO files
- represent outbound contracts only
- should be flattened enough for clients to consume without excessive round trips
- may expose nested summaries, but should not leak internal entity graph complexity
- must include IDs, status, audit metadata, and display-friendly labels when useful

E. Entity files
- model persisted business state only
- must extend shared base entity types where appropriate
- must use explicit table names and indexes via annotations where reasonable
- default to `FetchType.LAZY` for associations
- must not contain HTTP, GraphQL, or client presentation concerns
- use `@Version` for optimistic locking on heavily edited aggregates
- use embeddables/value objects for repeated concepts such as money, address, or period where justified

F. Repository files
- handle persistence and query composition only
- use Spring Data JPA for standard CRUD and filtered access
- use query repositories, JPQL, Specifications, or native SQL for complex reporting
- never hold business workflow decisions
- must be tenant-safe by construction

G. Service interface files
- define business capabilities, not technical helpers
- keep method names explicit: `createSale`, `closeShift`, `reconcileTank`, `runPayroll`
- expose transactional use cases, not repository-style verbs unless appropriate

H. Service implementation files
- own business invariants and orchestration
- validate status transitions and side effects
- publish events for secondary effects instead of doing everything inline
- coordinate repository calls, cache invalidation, and audit recording
- remain small enough to test at use-case level

I. Mapper files
- use MapStruct for deterministic DTO/entity mapping
- centralize custom conversions in shared mapper config or mapping utils
- never hide business decisions in mapper expressions
- prefer explicit mapping for money, enums, status summaries, and nested aggregates

J. Validation files
- contain domain-specific validation that Bean Validation annotations alone cannot express
- may validate cross-field rules, transition rules, or tenant-specific rules
- must be reusable by REST and GraphQL entry points

K. Query files
- hold read-optimized queries, projections, and materialized-view access
- are the correct place for analytics SQL, aggregation queries, and dashboard retrieval
- must not mutate transactional tables

L. Event files
- define internal domain events and event payloads
- support audit, notification, reporting refresh, and sync side effects
- should be immutable

M. Job files
- are scheduled entry points only
- must acquire a distributed lock before processing cluster-sensitive work
- must call services, not repositories directly, unless the job is purely infrastructure-focused
- must be idempotent and resumable

N. Config files
- wire beans, serializers, executors, scalars, interceptors, and security chains
- contain no business logic
- should be profile-aware

O. SQL migration and enhancement files
- must be additive and reversible where practical
- should never duplicate Hibernate-managed baseline tables without strong reason
- should focus on indexes, constraints, functions, triggers, views, materialized views, partitions, and automation helpers
- must separate shared/global schema logic from tenant schema logic

P. Test files
- every controller/resolver/service/query object added must have matching tests
- unit tests for business rules
- slice tests for web/graphql/repository behavior
- integration tests for multi-tenant flows, migration flows, cache behavior, and SQL object correctness

19. SPRING BOOT, JPA, GRAPHQL, AND LIBRARY USAGE RULES
====================================================

Spring Boot usage:
- use Spring Boot as the integration shell and dependency management backbone
- keep configuration strongly typed via `@ConfigurationProperties`
- keep startup deterministic: migrations first, seeders second, traffic last
- expose actuator health, readiness, metrics, and info endpoints from day one

Spring Data JPA / Hibernate usage:
- JPA owns baseline table generation from entities during development and controlled bootstrap flows
- production schema hardening happens through Flyway SQL after baseline structures exist
- use entities for transactional write models and authoritative state
- use repositories for aggregate access, not as service replacements
- default to lazy relationships and explicit fetch plans
- avoid massive bidirectional graphs unless they are truly needed
- use projections and query repositories for dashboard/reporting screens
- use batching for imports, sync replay, payroll generation, and report precomputation

GraphQL usage:
- add `spring-boot-starter-graphql`
- define schema files under `src/main/resources/graphql/`
- use schema-first contracts so mobile, web, and desktop teams can review the API clearly
- model top-level `Query` and `Mutation` types by business capability, not by database table
- keep mutations command-oriented: `createSale`, `closeShift`, `approveRequest`, `runPayroll`
- use cursor or stable offset pagination for list fields
- add custom scalars where needed:
  - `Date`
  - `DateTime`
  - `BigDecimal`
  - `JSON`
  - `Upload` when we choose GraphQL multipart support
- inject tenant, actor, role, and trace metadata into GraphQL context
- use DataLoader for:
  - user summaries
  - branch summaries
  - product summaries
  - permissions
  - report drill-down lookups

REST usage:
- keep REST for command-heavy and integration-heavy flows
- keep endpoint naming stable and version-aware
- support idempotency keys on critical writes such as sales posting, invoice payment capture, and tenant onboarding
- support bulk endpoints for sync and imports

Recommended library plan for this backend:
- `spring-boot-starter-graphql` for GraphQL API
- `flyway-core` and PostgreSQL Flyway support for shared + tenant schema migrations
- `spring-boot-starter-validation` for request validation
- `spring-boot-starter-data-redis` for cache, lock, and token metadata use cases
- `spring-boot-starter-actuator` + Micrometer Prometheus registry for observability
- `resilience4j-spring-boot3` or newer aligned module for retry, circuit breaking, and rate limiting integration
- `MapStruct` for mapping
- `ArchUnit` for architecture rule enforcement
- `Testcontainers` for PostgreSQL and Redis integration tests
- `springdoc-openapi` or the chosen Boot-compatible OpenAPI tool for REST documentation

Query strategy matrix:
- use JPA repository methods for CRUD and simple filtered searches
- use Specifications for admin filtering screens
- use JPQL/native query repositories for dashboard and reporting reads
- use materialized views for high-cost repeated summaries
- use SQL functions for recalculation and validation helpers
- use Redis only for cached or transient state, never as the source of truth for financial records

20. SQL AUTOMATION OBJECTS AND POST-HIBERNATE DATABASE HARDENING
====================================================

Hibernate/JPA gives us the baseline relational structure.
PostgreSQL automation objects give us production strength.

Naming conventions:
- functions: `fn_<domain>_<action>`
- triggers: `trg_<table>_<event>_<purpose>`
- views: `vw_<domain>_<purpose>`
- materialized views: `mv_<domain>_<purpose>`
- indexes: `idx_<table>_<columns>`
- unique indexes: `uq_<table>_<columns>`
- check constraints: `chk_<table>_<rule>`

Important rule:
Triggers must stay lightweight.
They should stamp metadata or enqueue refresh intent, not perform huge report recalculations inline.
Heavy recalculation should happen through jobs or explicit SQL functions invoked by services/jobs.

Cross-cutting SQL automation objects:

1. Timestamp automation
- `fn_common_touch_updated_at()`

- purpose: keep `updated_at` accurate across shared and tenant tables

- `fn_audit_record_change()`
- trigger pattern on sensitive tables:
  - sales
  - payments
  - invoices
  - payroll runs
  - approval requests
  - tenant lifecycle tables
- purpose: normalize audit payload capture for inserts, updates, and controlled deletes

3. Summary refresh queue marker
- `fn_reporting_mark_summary_stale()`
- trigger pattern: `trg_<table>_aiud_mark_summary_stale`

21. JAVA CONCURRENCY, THREADING, MULTIPROCESSING, AND SCALING RULES
====================================================

We will use Java concurrency intentionally, not cosmetically.

Important distinction:
- threading is inside one JVM
- concurrency is coordinated work happening at the same time
- true multiprocessing for this backend mainly means multiple JVM instances, worker containers, or pods processing work in parallel

Non-negotiable rules:
- never create raw threads manually inside business code
- never share `EntityManager`/Hibernate session across threads
- never parallelize transactional writes blindly
- never use async code to hide slow design
- every async path must be observable, retryable where appropriate, and safe to run more than once

Recommended execution model:

1. Request processing
- keep request thread work short and deterministic
- perform validation, authorization, tenant resolution, and core writes in the main transactional path
- return quickly for long-running tasks by creating a job record and processing asynchronously

2. Virtual threads / lightweight blocking concurrency
- where supported by the chosen Boot/runtime combination, use virtual-thread-friendly executors for high-volume blocking I/O tasks
- suitable for:
  - external API calls
  - email/SMS dispatch
  - report export fan-out
  - batched read orchestration
- do not combine virtual threads with uncontrolled database fan-out

3. Bounded thread pools
- define dedicated executors such as:
  - `ioExecutor`
  - `reportExecutor`
  - `notificationExecutor`
  - `syncExecutor`
- bound queue sizes and concurrency explicitly
- separate CPU-heavy work from blocking I/O work

4. CompletableFuture / async orchestration
- allowed in service orchestration and jobs
- useful for parallel read aggregation, for example:
  - branch summary + pricing + stock overview
  - payroll prefetch of attendance + leave + contracts
  - dashboard cards assembled from multiple read models
- each parallel task must own its own transaction boundary if it touches the database

5. Scheduled work
- use `@Scheduled` only with distributed locking
- coordinate cluster safety with Redis locks or PostgreSQL advisory locks
- log every job execution and duration
- jobs must support restart after crash without corrupting state

6. Event-driven secondary processing
- use internal events for:
  - audit recording
  - notification dispatch
  - cache invalidation
  - summary refresh requests
  - sync side effects
- keep primary transaction focused on authoritative write completion

7. CPU-heavy workloads
- PDF/Excel generation, compression, AI inference, and large exports must run on dedicated bounded executors or separate worker processes
- if a workflow becomes large enough, promote it to a dedicated worker deployment instead of overloading the API process

8. Batch processing
- use chunked processing for:
  - sync replay
  - payroll generation
  - stock recalculation
  - backfill reporting jobs
  - tenant bootstrap seeding
- combine batching with JDBC batch settings and careful transaction chunk sizes

9. Horizontal scale strategy
- scale the backend by running multiple application instances behind Nginx/load balancer
- keep session/token metadata centralized in Redis or database
- keep scheduled jobs singleton-per-cluster via lock coordination
- avoid local filesystem dependence for anything critical

10. When not to use parallelism
- simple CRUD flows
- status transition commands with heavy locking risk
- highly contended financial postings
- operations where ordering is the business guarantee

Examples of good concurrency use in PetroFlow:
- generating monthly branch performance summaries in parallel per branch with bounded workers
- dispatching notifications asynchronously after a committed transaction
- refreshing several dashboard materialized views in separate jobs
- processing device sync batches in parallel per device, but not interleaving batches for the same device unsafely

Examples of bad concurrency use:
- saving one sale and its payments from multiple threads inside one request
- reading lazy JPA entity graphs from async callbacks after the transaction is closed
- spawning a thread per incident/request/sale without executor limits

22. IMPLEMENTATION BATCH RULES FOR FUTURE FILE CREATION
====================================================

Whenever we create files from this point onward, every implementation batch should explicitly state:
- which package/module owns the change
- whether the file is shared-schema or tenant-schema relevant
- whether the file is REST-only, GraphQL-only, or shared by both
- what cache keys are affected
- what SQL enhancement objects are required
- what tests must be created with it
- whether async processing or scheduled refresh is involved

For each feature batch, think in this order:
1. request/contract shape
2. validation rules
3. service/use-case logic
4. persistence model
5. SQL hardening objects
6. cache strategy
7. audit/events
8. GraphQL read model if needed
9. tests
10. operational concerns

That is how this backend stays coherent while it grows into a real ERP super app core.
